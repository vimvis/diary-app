# HYROX Ticket Monitoring MVP

## 목표

회원이 웹사이트에 로그인해서 원하는 HYROX 티켓 조건을 등록하면, 서버가 정해진 시간에 상태를 확인하고 `매진 -> 구매 가능`으로 바뀌는 순간 이메일을 보내는 웹 서비스를 만든다.

초기 MVP는 `카카오톡` 대신 `이메일 알림`으로 시작한다.

## MVP 범위

- 회원가입 / 로그인
- 관심 티켓 등록 / 수정 / 삭제
- HYROX 인천 이벤트 1개 지원
- 하루 5회 자동 모니터링
- 구매 가능 상태 감지 시 이메일 발송
- 최근 감지 이력 확인
- 관리자용 체크 로그 확인

제외:

- 자동 구매
- 다중 티켓 플랫폼 지원
- 카카오 알림톡
- 회원별 자유 스케줄 설정
- 앱 푸시 알림

## 핵심 사용자 시나리오

1. 사용자가 회원가입 후 로그인한다.
2. 원하는 티켓 조건을 등록한다.
3. 서버는 매일 한국시간 `09:00`, `12:00`, `15:00`, `18:00`, `21:00`에 티켓 상태를 확인한다.
4. 조건이 `Sold out`에서 `Available`로 바뀌면 이메일을 발송한다.
5. 사용자는 이메일에서 즉시 구매 링크로 이동한다.

## 등록 예시

- 이벤트: `HYROX Incheon`
- 날짜: `2026-05-16`
- 요일: `토요일`
- 디비전: `Men Singles`
- 세부 타입: `Open`
- 알림 이메일: 회원 이메일 사용

## 상태 정의

티켓 상태는 아래처럼 단순하게 시작한다.

- `unknown`: 아직 확인 전
- `sold_out`: 매진
- `available`: 구매 가능
- `error`: 확인 실패

알림은 아래 경우에만 보낸다.

- 직전 상태가 `sold_out`
- 현재 상태가 `available`

아래 경우에는 알림을 보내지 않는다.

- 처음부터 `available`
- `available` 상태가 계속 유지
- 확인 실패

## 추천 기술 스택

### 프론트엔드

- `Next.js`
- `TypeScript`
- `Tailwind CSS`

### 백엔드

- `Next.js Route Handlers` 또는 `NestJS`
- 초기 MVP는 `Next.js` 단일 저장소가 단순하다.

### 데이터베이스

- `PostgreSQL`
- ORM은 `Prisma`

### 인증

- `NextAuth` 또는 이메일/비밀번호 기반 자체 인증
- MVP는 구현 속도를 위해 `NextAuth + Credentials` 또는 `Clerk/Supabase Auth`도 가능

### 모니터링 엔진

- `Playwright`
- HYROX 티켓 페이지가 동적 렌더링이면 브라우저 자동화가 가장 현실적이다.

### 스케줄러

- `cron`
- 배포 환경에 따라 `Vercel Cron + Worker API` 또는 `Railway cron` 사용

### 이메일 발송

- `Resend` 추천
- 대안: `SendGrid`, `AWS SES`

`Resend`를 우선 추천하는 이유:

- 연동이 빠름
- 템플릿 없이도 MVP 발송 가능
- 개발 경험이 단순함

## 권장 아키텍처

1. 사용자가 웹에서 관심 티켓 조건을 저장
2. 스케줄러가 3시간 간격으로 모니터링 작업 시작
3. 모니터링 워커가 HYROX 티켓 페이지에서 현재 상태 수집
4. 상태를 내부 기준 형태로 정규화
5. 등록된 사용자 조건과 매칭
6. 상태 변화가 있으면 DB 기록
7. `sold_out -> available`이면 이메일 발송

핵심 원칙:

- 같은 이벤트 페이지는 한 번만 수집하고 여러 사용자 조건에 재사용한다.
- 사용자마다 브라우저를 띄우지 않는다.
- 상태 변화 기반으로만 알림을 보낸다.

## 상태 판별 로직

초기에는 아주 단순하게 시작한다.

1. Playwright로 티켓 페이지 접속
2. 날짜와 디비전 조건 선택
3. 해당 항목의 버튼 또는 문구 읽기
4. 아래 규칙으로 상태 변환

- `Sold out`, `Unavailable` 유사 문구 -> `sold_out`
- 구매 버튼 활성화 -> `available`
- 항목을 찾지 못함 -> `error`

주의:

- 문구는 사이트 언어/구조 변경에 취약하다.
- 따라서 장기적으로는 텍스트 기반 + 버튼 활성 상태를 함께 확인해야 한다.

## 스케줄 정책

한국시간 기준 고정 실행:

- `09:00`
- `12:00`
- `15:00`
- `18:00`
- `21:00`

운영 규칙:

- 작업 중복 실행 방지 락 필요
- 이전 작업이 실패해도 다음 작업은 독립 실행
- 실패 시 관리자 로그에 남김

## 이메일 알림 설계

제목 예시:

- `[HYROX 알림] 인천 토요일 Men Singles Open 티켓 구매 가능`

본문 예시:

- 어떤 이벤트인지
- 어떤 날짜/종목인지
- 몇 시에 구매 가능으로 감지했는지
- 바로 이동할 수 있는 구매 링크
- 이 알림은 상태 감지 기준이며 실시간 재고 보장은 아니라는 안내

예시 문안:

```txt
HYROX Incheon 티켓 상태가 변경되었습니다.

- 이벤트: HYROX Incheon
- 날짜: 2026-05-16 (토)
- 종목: Men Singles
- 세부 타입: Open
- 감지 시각: 2026-03-19 15:00 KST

구매 페이지:
https://...

안내:
티켓 상태는 감지 시점 기준이며, 실제 접속 시 다시 매진되었을 수 있습니다.
```

## 관리자 운영 포인트

- 감시 실패 누적 시 알림

## MVP 개발 순서

### 1단계

- 프로젝트 초기화
- 로그인 / 회원가입
- DB 연결
- 기본 레이아웃

### 2단계

- 이벤트 / 티켓 옵션 등록 구조 설계
- 관심 티켓 등록 UI
- 마이페이지 목록 UI

### 3단계

- Playwright 수집기 작성
- 상태 정규화
- 수동 실행 테스트

### 4단계

- cron 연동
- 상태 변화 감지
- 이메일 발송

### 5단계

- 관리자 로그 화면
- 실패 재시도 처리
- 중복 알림 방지 검증

## MVP 성공 기준

- 회원이 관심 티켓을 직접 등록할 수 있다.
- 스케줄러가 하루 5회 자동 실행된다.
- 특정 조건의 상태를 안정적으로 읽어온다.
- `sold_out -> available` 전환 시 이메일이 1회만 발송된다.
- 최근 체크 이력을 관리자가 확인할 수 있다.
